Documentation¶

Start with Get Started. It takes a new user from choosing an environment to the first reviewed agent task without requiring knowledge of Heartwood's internal architecture.

Choose a Reading Path¶

You want to... Read these pages in order
Try Heartwood for the first time Get Started → Work with the Agent
Use the visual or notebook experience Get Started → Browser and Notebooks
Work in Terra Use Heartwood on Terra → Work with the Agent
Work on Stanford Carina Use Heartwood on Stanford Carina → Work with the Agent
Run without a hosted model Connect a Model → Run a Model Locally
Prepare an institutional deployment Deploy Heartwood → Platform Support and Validation → Security and Compliance
Understand or contribute to the system Product Scope → System Architecture → Contributing

If a setup does not behave as described, start with Troubleshooting. It explains the readiness states, the most common project and model problems, and what diagnostic information is safe to collect.

Start Here¶

  • Get Started with Heartwood explains when to use a container, native installation, or platform image and follows the first project through model setup and conversation.
  • Work with the Agent explains conversations, action review, session controls, resume, and audit export.
  • Troubleshooting explains project readiness, local runtime failures, interface routing, and safe diagnostic collection.

Interfaces¶

  • Browser and Notebooks explains visual setup, shared project state, local-model preparation, conversations, and Jupyter routing.

Models¶

  • Connect a Model explains the choices shown during model setup and how credentials are handled.
  • Run a Model Locally explains model weights, inference servers, recommended and user-selected Hugging Face downloads, automatic runtime selection, managed launch, and no-network validation.

Platforms¶

  • Choose Where to Run Heartwood compares workstation, container, Terra, Carina, and other managed environments.
  • Run Heartwood in a Container provides the shortest workstation setup and documents the available image variants.
  • Heartwood on Terra provides the custom-image, Jupyter, proxy, and synthetic validation workflow.
  • Heartwood on Stanford Carina provides the native installation and synthetic GPU workflow.

Deployment¶

  • Deploy Heartwood introduces artifact selection, persistence, model access, security boundaries, and validation claims.
  • Build a Platform-Specific Image defines how maintainers extend Heartwood from another platform base image.
  • Platform Support and Validation records release-specific evidence without implying institutional approval.
  • Release Heartwood defines the protected release process.

How Heartwood Works¶

These documents progress from product intent to implementation and validation details:

  • Product Scope defines the mission, users, boundaries, and reference workflow.
  • System Architecture defines component ownership, the project and state contract, and runtime data flow.
  • Security and Compliance defines threat boundaries and evidence requirements.
  • Skills and Extensions defines Skill packaging, trust, detection, and extension behavior.
  • Audit and Reproducibility defines in-boundary records, replay, and scrubbed export.
  • Platform Architecture defines deployment assumptions and target-environment rationale.
  • Testing and Evaluation defines test layers and model capability evidence.
  • Engineering and Releases defines repository, toolchain, continuous-integration, and supply-chain policy.

Reference¶

  • Project Files and State defines the current-directory project boundary and .heartwood/ layout.
  • Glossary and Acronyms explains specialized terms used throughout the project.
  • Contributing defines the contributor workflow and local checks.

Documentation Status¶

Term Meaning
Implemented Code and repository tests exist for the behavior.
CI-validated Automated checks exercise the behavior with synthetic data or controlled test infrastructure.
Live-validated A published artifact has passed the documented synthetic workflow in the real platform control plane.
Institution-approved The deploying institution approved the exact image, model route, credentials, data use, network controls, and evidence. Heartwood cannot assign this status.
Release-ready Every acceptance gate for the declared release scope is reproducible from an immutable artifact.

Project Planning¶

GitHub Issues own planned implementation, dependencies, and acceptance criteria. The Heartwood Project owns delivery status. Documentation describes current behavior and durable rationale; it does not serve as a backlog, implementation diary, or decision transcript.

Published Documentation¶

The public Heartwood documentation site is built only from published release tags. Each published documentation version remains available at its Semantic Version path. The /stable/ alias follows the latest stable release, while /preview/ follows the latest prerelease. The site root opens stable documentation once a stable version has been published; before then, it opens the current preview. Pull requests and main validate evolving source without changing any published page.

Preview the current source locally:

uv sync --locked --only-group docs
uv run --no-sync python deploy/stage_documentation.py
uv run --no-sync zensical serve

Use uv run --no-sync zensical build --clean --strict for the same strict build used by continuous integration.
