# Troubleshooting

Start from the directory Heartwood should treat as the project:

```bash
pwd
heartwood doctor
```

The diagnostic is read-only and checks project storage, model selection, credentials, route policy, and required compute.

## Readiness States

| State | Meaning | Next step |
|---|---|---|
| `ready` | The selected model and policy are available | Run `heartwood` or `heartwood serve` |
| `setup-required` | No complete model connection exists | Run `heartwood` and complete setup |
| `compute-required` | A downloaded model is selected but not running | Run `heartwood launch` or `heartwood launch --web` |
| `recovery-required` | A configuration, artifact, runtime, or platform check failed | Correct the failed check and rerun `doctor` |

`compute-required` is normal between local-model sessions.

## Project or Session Is Missing

Confirm that `pwd` identifies the expected project and that its `.heartwood/` directory is present. Starting in a parent or nested directory opens another project.

In a container, mount the same host directory at `/workspace`. On Terra, keep the project below `/home/jupyter` and retain the persistent disk. Use the same session identifier when switching interfaces.

## Model Setup Fails

List the current state:

```bash
heartwood models list
heartwood models validate
```

A provider token entered interactively belongs only to the running process. Enter it again after restart.

For another Hugging Face repository, inspect before download:

```bash
heartwood models inspect <owner/model>
```

Heartwood rejects unsupported families, custom code, ambiguous artifacts, and incompatible formats rather than guessing. The public download workflow does not cover every private or gated repository.

## Local Model Does Not Start

Downloading model files does not start the server. Run `heartwood launch` or `heartwood launch --web`.

Startup verifies the model, assesses memory, selects a context size, starts the server, and waits for readiness. Review the printed stage and:

```text
.heartwood/logs/local-model.log
```

The portable image uses CPU even when a GPU is attached. NVIDIA inference requires the explicit GPU image, a compatible driver, and enough GPU memory. Heartwood does not silently fall back after a GPU failure.

## Browser Does Not Open

Keep `heartwood serve` or `heartwood launch --web` running. On a local machine, open `http://127.0.0.1:8767/`.

On Terra, use the complete authenticated route printed by Heartwood or generated by the tutorial notebook. A shortened `/proxy/8767/` path returns 404. Use the terminal if the proxy is unavailable.

The notebook can replay a configured session but does not start a downloaded model. Keep `heartwood launch --web` running for notebook tasks that use local inference.

## Action Review Is Waiting

The active interface owns the pending decision. Do not approve the same turn from another process.

Heartwood applies one decision to the complete OpenHands action group. Reject the group when any command, file operation, path, or proposed content is unclear.

## Collect Safe Diagnostics

Before opening an issue, collect only:

- the Heartwood version and container or installer identifier;
- reviewed output from `heartwood doctor`;
- the failing command and exit status;
- the relevant error without prompts, credentials, participant data, or unrelated paths;
- the environment and model route in use.

Do not attach `.heartwood/`, full conversations, environment dumps, credentials, protected data, or broad directory listings. Remove sensitive content before using the [GitHub issue chooser](https://github.com/SchmiedmayerLab/heartwood/issues/new/choose).
